AutoCompiler: Automatic Repo-Level Compilation Tool Based on AI Agents

TL;DR: AutoCompiler is a tool that automatically compiles projects from their source code repositories. Going beyond our paper, this post will show some extended experiments focusing on the application of AutoCompiler to various downstream tasks.

Resources:

  • You can access the source code of this project at Ch3nYe/AutoCompiler.
  • The study of methodology behind this tool can be found in our paper:

    CompileAgent: Automated Real-World Repo-Level Compilation with Tool-Integrated LLM-based Agent System
    Hu Li, Ch3nYe(Co-first Author), et al. arXiv:2505.04254

Introduction

There are many scenarios where compilation of code repositories needs to be performed, such as CI/CD pipelines, automated fuzzing tests, code security checks, malware code audits, and so on. However, the compilation process can be time-consuming and error-prone, especially when dealing with large codebases or complex dependencies.

To address this issue, we propose AutoCompiler, the first automatic repo-level compilation tool based on AI agents. AutoCompiler leverages state-of-the-art AI models to analyze the source code of a project and build the code artifact automatically, as well as handling possible errors occured in compilation. This tool can significantly reduce the time and effort required to compile code repositories, making it easier to integrate compilation tasks into various workflows.

Currently, AutoCompiler are built for C/C++ projects those are more easily to arise compilation issues, and it is designed to be extensible to other programming languages. If anyone is interested in extending AutoCompiler to support other languages, please feel free to do it.

Challenges

At the very first, we need to identifiy the challenges of automatic compilation in code repositories. We thus conduct an extensive study on the compilation process of 100 popular C/C++ projects on GitHub. In general, the compilation process of code repositories is challenging due to the following reasons:

  • Challenge 1: The compilation process is highly variable and unstructured. Developers have significant freedom in designing the compilation process, making it fundamentally difficult to automate project-level compilation.
  • Challenge 2: Compilation guides are often poorly documented, with issues such as varying guide formats and styles, outdated information, and missing documentation. These problems prevent the automation of the compilation process through simple guide parsing.
  • Challenge 3: The compilation environment and dependencies are complex. Issues such as missing dependencies, inconsistently managed dependency sources, and conflicts between dependencies and the current environment frequently lead to compilation failures.

Methodology

Here, we just briefly introduce the methodology of AutoCompiler. For more details, please refer to our paper.

Figure 1: The overview of AutoCompiler.

Figure 1 illustrates the workflow of AutoCompiler from circled 1 to circled 16. Generally, our AutoCompiler employs a master agent to charge the whole compilation process, and calls various specialized tools to tackle different requirements. The reasoning skills and heuristics of LLM-based agents, along with their capability to understand unstructured input, make them well-suited for tasks like automated compilation, in contrast to traditional hard-coded automation programs. In other words, the LLM-driven approach fundamentally mitigate the Challenge 1.

There are two main modules:

  • CompileNavigator Module: locates the correct compilation instructions within a code repository by employing tools like Shell, File Navigator, and Instruction Extractor. Its role is to accurately identify files containing compilation instructions and extract those instructions to facilitate the compilation process. Designed to mitigate Challenge 2.
  • ErrorSolver Module: addresses compilation errors during the build process by utilizing tools such as Website Search and Multi-Agent Discussion. It aims to resolve errors like missing dependencies or configuration conflicts by discussing and consensus-building among multiple agents or by searching for solutions online. Designed to mitigate Challenge 3.

Evaluation

We conducted experiments to evaluate the performance of AutoCompiler in practical scenarios, including fuzzing tests, code security checks, and malware code audits. And the backend LLM is GPT-4o for the following experiments.

Overall Effectiveness

For the overall tests, we collect 100 popular C/C++ GitHub projects, which has 14 topics ranging from crypto to networking. We collected the software artifacts for these projects from software sources, from which we obtained the compilation tagets. The compilation was considered successful if the targets were built.

We include the OSS-Fuzz-Gen as the only baseline, which is a state-of-the-art fuzzing tool that automatically generates fuzz tests for C/C++ projects. More specifically, the oss-fuzz-gen/experimental/c-cpp provide 14 different auto-build-heuristics base on the files existed in target repo. For example, if the repo has a Makefile, the heuristic with a make command will be used to build the project.

  • default-not-static: the project use a default building configuration without static library.

Downstream Task: code security checks

Many code security checks are done in or after the compile-time. When the number of code repository waiting for check scaled up (up to hundreds or thousands), manual compilation becomes infeasible.
Therefore, we also conduct experiments on including code security checks in the compilation process with the AutoCompiler:

  • gcc-sanitizer: gcc built-in code sanitizer, using with -fsanitize=address -fsanitize=leak flags.
  • scan-build: the code static analyzer comes up with clang, wrapper building command with scan-build to perform static analysis.

Discussion and Future Wpork

Extension to More Languages

Extending AutoCompiler to support more compiled programming languages is a natural progression. The current implementation focuses on C/C++ projects, but the underlying principles can be applied to other languages with similar compilation processes.

Extension to Multiple Architectures

In high-level programming language ecosystems (such as C/C++/Rust/Go), cross-architecture compilation toolchains are often available, allowing the same source code to be compiled into executables for different architectures to run on various machines. However, some projects use low-level architecture-specific code to enhance performance/security, which hinders their multi-architecture scalability. Considering these factors, our evaluation did not perform multi-architecture compilation for the selected projects.

Fine-Grained Compilation Configuration

In some scenarios, we need to set compiler parameters or use compilation options beyond the default configuration to build different binaries. For example, during software release, stripping and Os parameters need to be specified, while during debugging, we need binaries with debugging information. Additionally, enabling dynamic compilation, special algorithm implementations, or compiling different distributions may require configurations related to specific user needs, such as compiling static libs for fuzzing.
While we did not conduct experiments on AutoCompiler with different configurations due to the inability to exhaust all these needs, we have open-sourced AutoCompiler and rely on the community to test it in various configurations.

Refine Compilation Steps

AutoCompiler generates many trial-and-error steps during the compilation process. These steps are erroneous branches in the compilation workflow, causing additional overhead and reducing compilation efficiency. Taking measures to prune erroneous actions during the compilation process would be beneficial for maintaining a feasible compilation orchestration, avoiding the inclusion of incorrect commands in the replay. Such optimizations do not affect the current effectiveness of AutoCompiler but merely refine the automatic compilation results. Therefore, we consider these optimizations as future work.

Conclusion

In this post, we introduced AutoCompiler, the first automatic repo-level compilation tool based on AI agents. Extensive experiments demonstrate the effectiveness and practicality of AutoCompiler in various downstream tasks. We hope that AutoCompiler will be a valuable tool for automating compilation tasks in various scenarios, making it easier to integrate compilation tasks into various workflows.
